前端VUE项目开发规范

规范的目的是为了编写高质量的代码，让团队成员维护项目时少一点头疼点，大家在一起开发是快乐的。

引自《阿里规约》的开头片段：

—-现代软件架构的复杂性需要协同开发完成，如何高效地协同呢？无规矩不成方圆，无规范难以协同，比如，制订交通法规表面上是要限制行车权，实际上是保障公众的人身安全，试想如果没有限速，没有红绿灯，谁还敢上路行驶。对软件来说，适当的规范和标准绝不是消灭代码内容的创造性、优雅性，而是限制过度个性化，以一种普遍认可的统一方式一起做事，提升协作效率，降低沟通成本。代码的字里行间流淌的是软件系统的血液，质量的提升是尽可能少踩坑，杜绝踩重复的坑，切实提升系统稳定性，码出质量。

(一)文件命名规范

1.1 项目命名

采用首字母大写单词隔开

My-Pro

1.2 JS、CSS、SCSS、HTML、PNG 文件命名

全部采用小写方式， 以中划线分隔

正例： render-dom.js / signup.css / index.html / company-logo.png

反例： renderDom.js / UserManagement.html

1.3 目录命名

为了达到目录清晰,列出下面常用目录,以此规范命名.

以下仅列出需要配置的目录和文件

  • 一级目录
    • src 项目主入口
    • public 静态资源存放文件,项目全局公共资源文件,图片|字体|icon等,一般存放项目logo,网页角标ico图片
    • .env.development 开发环境变量(一般只需要写开发环境的ip请求地址)
    • .env.production 生产环境变量(生产环境的ip请求地址)
    • vue.config.js vue项目配置文件(一般无需特别配置,vue有一套现成的loader)
  • 二级目录src
    • api 全局接口文件目录
    • assets 统一资源目录文件目录
    • components 全局公共组件目录
    • icons icon图片,按需使用
    • layout 布局页面目录
    • mixins 全局公共混入的代码目录
    • route 路由配置目录
    • store vuex配置目录
    • styles 全局css,less,sass,scss目录
    • utils 全局方法,工具目录
    • plugins 插件详细配置目录
    • views 视图目录
    • App.vue 项目入口
    • main.js 全局配置文件
  • 区分目录功能,不要在main.js主要配置中写逻辑代码

1.4 特殊目录下文件命名

  • api,view,router中文件,组件名必须统一,一一对应,方便维护
    • @/views/test.js
    • @/views/test/test.vue
    • @/route/modules/test.js
  • 路由中组件命名和页面命名统一(cv的事)
  • 一个view页面对应一个api(同名js文件)
  • router中一个模块单分出一个文件导出(根据路由视图分解成模块化)
  • plugins目录中对于外部插件的单独配置(element,等)

1.5组件命名

采用首字母大写单词隔开

My-Component

全局组件命名

以All+组件名标记全局组件,做到一眼清

All-组件名

1.6 组件文档

能够全局复用的组件也不多,每个全局组件都需要有对应的文档,方便后续维护及优化

文档格式

  • 组件
    • 组件代码
    • 参数说明
  • 使用方法
    • 逻辑部分使用mixins混入,不直接写入调用页面
    • 传参说明

HTML5 中新增很多语义化标签，所以优先使用语义化标签，避免一个页面都是 div 或者 p 标签

1.2. css规范

1)按以下顺序组织

1、@import;
2、变量声明;
3、样式声明;

2)重复样式的抽离提取为公共样式

抽离出通用性强,使用频繁的css语句,如:

  • 左右居中
  • 上下居中
  • 上下左右居中
  • 清除浮动
  • margin,padding的常用值
  • ….

3)关于主题样式抽离为css变量方便后续修改维护

如侧边栏,表格样式等

4) css文件的模块化

正常项目中肯定不止一个css样式文件,以styles目录下的index文件为出口,在index.scss中 @import导入其他css配置文件

styles/index.scss

最后在main.js中引入index.scss即可

1.3. Javascript 规范

1) 使用 2 个空格进行缩进

2) 统一使用单引号

api接口方法中需要字符串拼接接口时,使用es6模板字符串语法

3) 优先使用es6,es7中新增的语法糖和函数

比如箭头函数、await/async ， 解构， let ，const , for…of , …(展开运算符) 等等

4) 三元运算符

  • 不要写太长的三元运算符!!!
  • 简单明了的逻辑判断使用三元运算符
  • 复杂判断方法使用计算属性
  • 三元运算符不可超过三层.
  • 多层if判断使用switch,更清晰明了.

5) console.log使用完毕后立马注释或者删除.

后续按需求统一

1.5. mixins规范

1)变量命名

因为是可以到处拿来即用的代码,为了避免冲突以及和其他代码块区分开来,变量命名为:

'Mixin' + '变量名'

全局通用逻辑不多,变量命名越详细越好

2)代码块需要注释用途

这种跨文件的复用文件都需要注释.

1.6. route规范

1)区分以下三个模块

  • 路由配置模块
    • index.js
  • 路由守卫模块
    • permission.js
  • 具体路由模块
    • modules目录
    • 具体模块.js,一般为一个侧边栏导航部分

2) 具体配置

1.7. vuex规范

1)不要为了使用vuex而去用vuex!!!

2)列出以下几种场景

  • 跨页面传递状态

  • 多个页面或者组件中操作同一的数据,使用vuex管理状态比较方便

  • 记录用户操作,如,弹框确认后不再显示等

1.8. 环境变量

1)开发环境变量(本地)

.env.development文件

2)生产环境变量(打包上线)

.env.production文件

3)主要是配合axios使用

使用webpack的api调用

当后端需要更换地址时更改.env.production中的VUE_APP_BASE_API就行了,这就是为什么上面需要全局使用统一的请求方法,使用环境变量应对这种情况会非常方便

1.9. 通知消息规范

1)网络请求

关于网络请求统一使用Message展示请求状态

  • 请求成功
  • 请求失败
  • 请求超时
  • ….

2)用户操作通知

  • 操作成功
  • 删除成功
  • 不可删除
  • …
